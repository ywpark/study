# 스레드

---

## 스레드란 무엇인가

- `스레드 (Thread)`는 프로세스 안에서 실행되는 흐름의 단위이다.
- 하나의 프로세스는 하나 이상의 스레드를 가질 수 있다.
- 프로세스가 자원 소유의 단위라면, 스레드는 실제 실행의 단위라고 볼 수 있다.

## 프로세스와 스레드의 차이

- `프로세스`
  - 독립된 메모리 공간을 가진다.
  - 다른 프로세스와 자원을 기본적으로 공유하지 않는다.
  - 생성과 문맥 교환 비용이 상대적으로 크다.
- `스레드`
  - 같은 프로세스 내부의 자원을 공유한다.
  - 생성과 전환 비용이 프로세스보다 작다.
  - 공유 자원이 많기 때문에 동기화 문제가 더 쉽게 발생한다.

## 스레드의 구성 요소

- 각 스레드는 다음과 같은 독립적인 실행 정보를 가진다.
  - 스레드 ID
  - 프로그램 카운터
  - 레지스터 값
  - 스택
- 반면 같은 프로세스의 스레드들은 다음 자원을 공유한다.
  - 코드 영역
  - 데이터 영역
  - 힙 영역
  - 열린 파일

## 멀티프로세스와 멀티스레드 비교

- `멀티프로세스`
  - 안정성이 높다.
  - 하나의 프로세스 오류가 다른 프로세스에 미치는 영향이 상대적으로 작다.
  - 프로세스 간 통신 비용이 크다.
- `멀티스레드`
  - 자원 공유가 쉬워 협업이 빠르다.
  - 응답성과 자원 활용성이 좋다.
  - 한 스레드의 오류가 프로세스 전체에 영향을 줄 수 있다.

## 사용자 수준 스레드와 커널 수준 스레드

- `사용자 수준 스레드 (User-Level Thread)`
  - 사용자 라이브러리 수준에서 관리되는 스레드
  - 커널이 각 스레드를 직접 알지 못할 수 있다.
  - 전환이 빠르지만, 하나가 시스템 콜로 막히면 전체가 영향을 받을 수 있다.
- `커널 수준 스레드 (Kernel-Level Thread)`
  - 운영체제가 직접 관리하는 스레드
  - 스케줄링과 병렬 실행 지원이 유리하다.
  - 관리 비용은 사용자 수준 스레드보다 더 크다.

## 멀티스레드의 장점

- `응답성 향상`
  - 한 스레드가 오래 걸리는 작업을 수행하는 동안 다른 스레드는 사용자 요청에 응답할 수 있다.
- `자원 공유 용이`
  - 같은 메모리 공간을 사용하므로 데이터 교환이 효율적이다.
- `경제성`
  - 프로세스 생성보다 스레드 생성 비용이 적은 경우가 많다.
- `병렬성 활용`
  - 멀티코어 환경에서 여러 스레드를 동시에 실행할 수 있다.

## 멀티스레드의 문제점

- `경쟁 조건 (Race Condition)`
  - 여러 스레드가 공유 자원에 동시에 접근하면서 결과가 달라지는 문제
- `동기화 비용`
  - 자원을 안전하게 보호하기 위해 락, 세마포어 같은 기법이 필요하다.
- `디버깅 어려움`
  - 실행 순서가 매번 달라질 수 있어서 재현이 어렵다.

## 스레드와 동기화의 관계

- 스레드는 자원을 공유하기 때문에 프로세스보다 동기화 문제가 더 직접적으로 발생한다.
- 따라서 멀티스레드 프로그램에서는 `임계 구역`, `상호 배제`, `데드락`을 함께 이해해야 한다.

## 정리

- 스레드는 프로세스 안에서 실행되는 흐름의 단위이며, 멀티스레드는 성능과 응답성 측면에서 큰 장점을 준다.
- 하지만 공유 자원으로 인해 동기화 문제가 쉽게 발생하므로, 스레드는 반드시 동기화와 함께 학습해야 한다.
