# 페이징을 통한 가상메모리 관리

---

## 기본정보

- 프로세스 연속 메모리 할당의 문제점
    - 외부 단편화
    - 물리 메모리보다 큰 프로세스 실행 불가
- `가상 메모리 (Virtual Memory`)
    - 실제 물리적 메모리(RAM)의 크기에 관계없이, 프로그램이 사용할 수 있는 메모리 공간을
    확장하는 기술

## 가상 메모리 관리 기법

- `페이징 (Paging)`
    - 가상 메모리와 물리적 메모리를 `페이지` 라는 작은 고정 크기의 블록으로 나누는 방식
        - `페이지 (Page)`
            - 가상 메모리의 일정 크기 단위로 나눈 블록
        - `프레임 (Frame)`
            - 물리 메모리를 일정 크기 단위로 나눈 블록
            - 페이지와 크기가 동일
    - 페이지를 프레임에 할당하는 가상 메모리 관리 기법
    - `스와핑 (Swapping)`
        - 페이지 단위로 진행
        - 프로세스를 실행하기 위해 모든 페이지가 적재될 필요 없음
            - 물리 메모리보다 큰 프로세스도 실행 가능
    - `페이지 테이블 (Page Table)`
        - CPU 입장에서 프로세스를 이루는 페이지가 어느 프레임에 적재되어 있는지 알기가 어렵고, 불연속적으로 배치되어 있다보니 순차적으로 실행 불가능
            - 이 문제를 해결하기 위한 것이 `페이지 테이블`
        - 프로세스의 가상 주소와 실제 물리 주소를 매핑하는 테이블
        - 프로세스마다 페이지 테이블이 존재
        - 물리적으로는 분산되어 있어도 CPU 입장에서 바라본 논리 주소는 연속적으로 보임
    - `내부 단편화 (Internal Fragmentation)`
        - 외부 단편화 문제는 발생하지 않음
        - 내부 단편화 문제가 발생
            - 마지막 프레임의 크기가 물리 메모리의 프레임 크기로 나누어 떨어지지 않는 경우
            - 발생하더라도 지극히 작은 크기로 발생됨
    - `프로세스 테이블 베이스 레지스터 (Process Table Base Register, PTBR)`
        - 프로세스의 페이지 테이블이 적재된 주소를 저장
        - 메모리 접근 시간이 2배
            - 페이지 테이블 접근
            - 프레임 접근
    - `TLB (Translation Lookaside Buffer)`
        - 메모리 접근 횟수를 줄이기 위한 기법
        - 페이지 테이블 정보를 캐시
    - `페이징 시스템의 논리 주소`
        - `페이지 번호 (Page Number)`
            - 가상 주소가 어느 페이제 속하는지 나타냄
        - `페이지 오프셋 (Page Offset)`
            - 해당 페이지에서의 특정 위치
    - `페이지 테이블 엔트리 (Page Table Entry, PTE)`
        - 페이지 테이블의 각각의 행
        - 구성
            - `유효비트 (Valid Bit)`
                - 해당 페이지 접근 가능 여부
                    - 0 : 페이지가 메모리에 배치 안됨
                    - 1 : 페이지가 메모리에 배치 됨
                - `페이지 폴트 (Page Fault)`
                    - 유효 비트가 0인 페이지로 접근하려고 하면 발생
                - 처리 순서
                    1. CPU가 기존 작업 백업
                    2. 페이지 폴트 처리 루틴 실행
                    3. 페이지 처리 루틴은 원하는 페이지를 메모리로 가져온 뒤 유효비트를 1 로
                    변경
                    4. 페이지 폴트 처리했다면 CPU는 해당 페이지에 접근
            - `보호비트 (Protection Bit)`
                - 페이지 보호 기능을 위해 존재
                    - 0 : 읽기와 쓰기가 모두 가능한 페이지
                    - 1 : 일기만 가능한 페이지
                - 페이지에 접근 할 권한을 제한하여 페이지를 보호
            - `참조비트 (Reference Bit)`
                - CPU가 해당 페이지에 접근한 적이 있는지 여부
                    - 0 : CPU 가 접근한 적 없음
                    - 1 : CPU가 읽거나 쓴 페이지
            - `수정비트 (Modified Bit)`
                - `Dirty Bit` 라고도 불림
                - 해당 페이지에 데이터를 쓴 적이 있는지 수정 여부
                    - 0 : 수정된 적 없음
                    - 1 : 수정된 적 있음
                - `왜 존재할까?`
                    - 수정된 페이지는 스왑 아웃될 때 보조기억장치에도 쓰기 작업을 거쳐야 하기
                    때문에
- `세그멘테이션 (Segmentation)`
    - 프로그램을 논리적으로 관련 있는 여러 세그먼트로 나누어 메모리를 할당하는 방식
    - 세그먼트는 크기가 가변적임
    - 세그먼트는 `코드`, `데이터`, `스택`, `힙` 등과 같은 논리적인 단위로 나누어짐

## 페이징 이점

- `fork()`
    - 프로세스는 기본적으로 자원을 공유 안함
    - 부모 프로세스가 적재된 별도의 공간에 자식 프로세스를 통째로 복제
        - 프로세스 생성 시간 지연
        - 메모리 낭비
- `쓰기 시 복사 (Copy On Write, COW)`
    - `fork()` 시 발생되는 Overhead 를 줄일 수 있음
    - 부모와 자식 프로세스가 동일한 프레임을 가리키도록 함
    - 부모/자식 프로세스 중 하나가 페이지에 쓰기 작업 수행 시 해당 페이지를
    별도의 공간으로 복제
        - 프로세스 생성 시간 절약
        - 메모리 절약

## 계층적 페이지 테이블 (Hierarchical Page Table)

- `Multilevel Page Table` 이라고도 함
- 프로세스를 이루는 모든 페이지 테이블 엔트리를 항상 메모리에 유지하지 않을 방법
    - 전체를 메모리에 유지하는 것은 메모리 낭비
- 페이지 테이블을 페이징하여 여러 단계의 페이지를 두는 방식
- 논리 주소 구성
    - `최상위 페이지 테이블 (Out Page Table)`
    - `하위 페이지 테이블 (Inner Page Table)`
    - `오프셋 (Offset)`
- 계층 단계는 여러 단계가 될 수 있음
    - 계층이 많다고 좋은 것은 아님
    - 메모리 접근 횟수가 증가하기 때문에 성능에 이슈가 발생할 수 있음
